# 栏目格式与短示例

> 📄 **MD 文件模式（默认）**：使用 `# H1 / ## H2 / ### H3` 标题，表格/代码块均可用。
> 📱 **Telegram 模式（备选）**：如下规则（无 `#` 标题/无表格/无代码块）。

## Telegram 模式专有格式限制

不要使用 `# / ## / ###` 标题，不要使用 `---` 分隔线，不要使用 Markdown 表格，不要整篇代码块。栏目标题统一写成 `🔥 **栏目名**` 这种形式。

## MD 文件模式格式对照

所有以下 Telegram 专有格式（`>` 引用块、`🔥 **粗体标题**`、列表字段替代表格）在 MD 文件模式下全部替换为常规 Markdown：用 `## N级标题` 替代 `🔥 **粗体**`，用表格替代列表字段，无需 `>` 引用块包装。内容字段（📌事件/🌊影响/👀后续/💡解读/✅可信度）保持不变。

## 通用单条新闻

• 🔴 [标题](URL)
  来源：平台#排名 · 热度 · 时间 · 类型
  📌 事件：一句话说明发生了什么。
  🧭 背景：复杂/重大事件才写。
  🌊 影响：影响谁、影响什么行业或决策。
  ⚠️ 风险：复杂/高风险/单源事件才写。
  👀 后续：下一步看什么。
  💡 为什么重要：一句话解释价值。
  ✅ 可信度：高/中/低；单源必须写“单源待核实”。

## 🧭 今日主线

• 🔴 主线名：[代表新闻](URL)

  后续：下一观察点。

要求：只放 3-6 条真正主线，不把普通热搜塞进主线。

## 🐙 GitHub 趋势

🐙 **GitHub 趋势**

要求：
- **数量**：日榜前 10 + 周榜前 10（去重后约 15-20 个），必须写 `日榜#N` / `周榜#N`
- **Stars**：必须写真实 Stars 数值。优先用 workspace 桥接脚本获取，不要写 `Stars未获取`
- **每条必须包含完整字段**：

• 日榜#N [owner/repo](https://github.com/owner/repo)
  ⭐ 18.4k · 语言：Python
  📌 项目：一句话说明项目做什么。
  🧩 是干嘛的：详细解释——**解决什么问题、适合谁用、成熟度如何、和同类相比有什么特点**。不能只写一句话。至少要写 2-3 句，让读者理解这个项目的核心价值。
  🔥 为什么热：说明排名、Stars、近期增长、社区讨论、发布节点或赛道热度。
  👀 后续看：文档完善度、安装体验、issue 活跃度、是否有真实案例、许可证。

- 项目简介未获取时写“项目简介未获取，需点开 README 核实”，不要假装知道。
- 如果用了 workspace 桥接获取 Stars，在 📊 平台统计中注明。

## 📜 论文·学术

📜 **论文·学术**

• 📜 [论文标题](https://arxiv.org/abs/0000.00000)
  arXiv：有 · 代码：未获取 · 时间：YYYY-MM-DD
  核心贡献：**用一句通俗易懂的话说清「这篇论文做了什么、解决了什么问题」**。不要只写专业术语，要考虑非该领域读者也能理解。
  工程价值：说明「谁用得上、实际落地可能性、是否对工程有启发」。
  局限：数据、复现、成本或适用范围。
  💡 解读：和已有方法相比改了什么、是否只是指标提升、对工程是否有启发。

要求：
- **核心贡献必须写清楚这篇论文是干什么的**，不能只用专业术语堆砌。至少要写 2 句话：第一句概括做什么，第二句解释为什么重要。
- **不写空泛评价**（如"意义重大""影响深远"）。
- 工程价值要具体：是降低训练成本、提高推理速度、还是开了新方向？
