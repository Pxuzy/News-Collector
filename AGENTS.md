## CodeGraph

在已由 CodeGraph 建立索引的仓库中，也就是仓库根目录存在 `.codegraph/` 时，需要理解或定位代码前，先使用 CodeGraph，再使用 `grep`、`find` 或直接读文件：

- **MCP 工具**（可用时）：`codegraph_explore` 通常可以一次回答大多数代码问题，返回相关符号的原始源码以及它们之间的调用路径。`codegraph_node` 可以返回单个符号的源码和调用方，也可以按行号读取整个文件。如果工具已列出但处于延迟加载状态，通过工具搜索按名称加载。
- **Shell**（始终可用）：`codegraph explore "<symbol names or question>"` 和 `codegraph node <symbol-or-file>` 会输出同类信息。

如果没有 `.codegraph/` 目录，完全跳过 CodeGraph。是否建立索引由用户决定。

# News-Collector 代理协作说明

这个仓库的目标是建设一个多源新闻采集代码库：从不同平台采集新闻，存入数据库，再让 skill 调用仓库程序或查询数据库完成新闻汇总整理。

## Git 管理

- 如果仓库还没有初始化 git，可以先执行 `git init`，让后续改动可追踪。
- 修改前先看 `git status --short`，区分用户已有改动和本次改动。
- 可以修改文件，但提交前必须先征得用户明确允许。
- 如需执行 `git commit`，提交信息使用中文，并清楚说明本次变更内容和验证结果。
- 未经用户明确允许，不要执行 `git commit`、`git push`、`git reset`、`git checkout --` 或其他会提交、发布、覆盖、回滚改动的操作。

## 代码查看

- 可以使用 CodeGraph 和 codebase-memory 辅助理解代码库。
- 如果仓库根目录存在 `.codegraph/`，需要理解或定位代码时优先使用 CodeGraph，再用 `rg`、文件阅读等常规方式补充。
- 如果需要了解项目历史决策、已知问题或跨会话上下文，可以先查 codebase-memory，再回到当前代码验证。
- 结论必须以当前仓库文件和实际命令结果为准，不要只凭记忆或旧文档判断。

## 工作流程

- 用 gstack 做完整开发周期：理解目标、定位代码、制定小步修改、实施、验证、汇报。
- 变更应保持窄范围，优先修真实入口、数据入库、查询接口和 skill 可调用路径。
- 验证要使用仓库里的实际命令；如果命令失败，说明失败原因和下一步，而不是假装通过。

## 测试工具

- 可以合理使用其它插件和 MCP 服务做测试、调试和结果核验。
- 使用插件或 MCP 服务前先判断是否和当前任务直接相关；优先选择能验证真实行为的工具，例如数据库查询、接口测试、浏览器测试、截图检查或结构化代码检索。
- 不为了形式化测试而堆叠无关插件。工具输出必须回到当前代码和实际结果上验证。
- 如果测试需要联网、启动外部程序、访问用户浏览器、写入工作区外路径或执行高风险操作，必须先按权限规则请求用户允许。

## 范围

- 暂时不需要额外的复杂流程、长篇架构说明或泛化规范。
- 新规则必须服务于当前目标：多源新闻采集、数据库沉淀、skill 查询和新闻汇总。
